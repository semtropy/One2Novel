# One2Novel 创作流水线 UX 审计报告

## 流水线总览：7 步

```
StartPage → [Step1 故事核心] → [Step2 参考分析] → [Step3 架构选择] → [Step4 角色阵容] → [Step5 章节蓝图] → [Step6 定位校准] → 进入写作
```

### 🔴 问题1：为什么需要上传两次参考书？

**原因**：StartPage 上传后，Step2（ReferenceDomain）仍会显示一个上传区。这是因为 Step2 的 `loadStats()` 在 mount 时异步加载，在加载完成前 `stats === null`，所以用户会短暂看到上传区。如果 API 调用失败（例如服务器还没完全就绪），`stats` 保持 null，用户会看到一个永久的上传提示。

**实际上**：如果 StartPage 上传成功，Step2 应该在加载完成后直接显示分析面板。但由于 `loadStats()` 静默吞掉错误（`catch { setStats(null) }`），用户无法区分"还在加载"和"加载失败要我重传"。

### 🔴 问题2：「确认架构」按钮是什么意思？怎么没有任何反馈？

**按钮实际行为**：
1. 保存回环定义 → `PUT /loop-definition`
2. 执行架构确认 → `POST /pipeline/step/architecture`（生成回环骨架 + 保存金手指）
3. 刷新页面数据 → `refetch()`

**反馈缺失**：
- 点击后按钮短暂显示 spinner，然后恢复
- 成功时：**没有任何视觉反馈** — 无绿色勾、无 toast、无"架构已确认"
- 失败时：在按钮下方显示一行红色小字（`saveError`），但位置在页面最底部，用户很可能看不到
- 按钮文案"确认架构"对新手用户完全无意义 — 不理解在确认什么

### 🔴 问题3：「生成模式」看不懂，「展开为章节」失败无反馈

**「生成模式」切换**：两个按钮"一键全生成"和"逐卷生成"并列，没有解释：
- 「一键全生成」= 调用 `POST /blueprint`，一次性生成所有卷+章（可能需要 30-60 秒）
- 「逐卷生成」= 先生成回环骨架，然后逐卷点击"展开为章节"
- 用户不知道它们在底层做什么，也不理解差异

**「展开为章节」失败无反馈**：
- `handleExpandVolume` 的 catch 块是空的：`catch {}`
- 如果 LLM 调用超时（120秒）或失败，按钮只是从"展开中..."变回"展开为章节"
- 用户看到按钮恢复但没有任何新章节出现，完全不知道发生了什么

**回环阶段标签**：每张卡片列出 trigger/enter/explore/setback/turn/climax/settlement 但只用颜色标注，没有解释这些英文 label 的中文含义。

### 🔴 问题4：回环迭代表为什么只有 100 多章？

**代码分析**：
```
// loopTemplateService.ts
const estimatedTotal = novel.estimatedChapterCount || 333  // 默认 333 章
const loopCount = Math.max(3, Math.min(12, Math.round(estimatedTotal / 10)))  
// = Math.min(12, Math.round(333/10)) = Math.min(12, 33) = 12  → ❌ 被硬上限卡住了
```
然后 LLM 生成的每轮回环章数范围是 6–25，平均约 12 章。
12 回环 × 平均 12 章 = 约 144 章。

144 章 × 3000 字/章 = 432,000 字 — 不到百万字的一半。

**根因**：`Math.min(12, ...)` 硬上限 12 轮回环。如果要达到 333 章（百万字），按每轮 12 章算需要 28 轮回环，但上限只允许 12。

### 🔴 问题5：定位校准看不懂，无法编辑，不知道要操作什么

**商业定位全部只读**：
- 题材、目标读者、核心卖点、架构标签、前30章承诺书 — 全部以 `<span>` 展示
- 没有任何编辑按钮、输入框
- 这些是 AI 自动生成的（`POST /framing`），但用户无法修改
- 架构标签（如"策略搭配型·御兽流"）是硬编码的映射表产物，不可更改

**「确认定位，校准期待参数」按钮**：
- 点击后调用 `POST /pipeline/step/calibration`，内部执行：
  1. 如果无 targetAudience → 重新生成 bookFraming
  2. 如果无 expectationProfile → 从架构模板构建默认值
- 但这些操作对用户完全不可见
- 成功/失败均无反馈

**爽点配方/钩子密度/伏笔窗口**：这些是从架构模板自动派生的参数，以柱状图/数值展示，但用户无法调整各类型的比例。

## 需要用户决策

这些改动涉及**前端 UX 重设计**，而非简单的代码修复。在动手之前，请确认优先级：

1. **修复回环数上限**（问题4）：改一行代码即可（`Math.min(12, ...)` → 放开上限），但会影响 LLM 调用次数
2. **补全所有按钮的加载/成功/失败反馈**（问题2+3+5）：每个按钮加 toast 或状态指示
3. **商业定位改为可编辑**（问题5）：需要新增 inline edit 组件
4. **Step2 上传检测优化**（问题1）：`loadStats()` 加 loading 状态，区分"加载中/无数据/加载失败"
5. **中文标签替换英文 key**（问题7+8）：`trigger→触发事件` 等映射已有但未使用

要继续执行修复吗？从哪个问题开始？
